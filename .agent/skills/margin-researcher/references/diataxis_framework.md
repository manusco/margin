# The Diataxis Framework (Documentation Strategy)

> "Documentation is not a single thing. It is four things."

The Diataxis framework splits technical content into four distinct quadrants based on two axes: **Action vs. Information** and **Learning vs. Working**.

## 1. Tutorials (Learning-Oriented / Practical)
*   **Goal**: Getting started.
*   **Style**: A lesson. Teacher/Student relationship.
*   **Structure**: "First do this, then do that. See? It works."
*   **Example**: "Build your first Blog in 10 minutes."
*   **Rule**: No abstractions. Concrete steps only.

## 2. How-To Guides (Problem-Oriented / Practical)
*   **Goal**: Solve a specific problem.
*   **Style**: A recipe.
*   **Structure**: "If X happens, do Y."
*   **Example**: "How to fix CORS errors."
*   **Rule**: Assume context is already known. Get to the fix.

## 3. Explanation (Understanding-Oriented / Theoretical)
*   **Goal**: Illumination. Context.
*   **Style**: A textbook.
*   **Structure**: "This is why X works this way."
*   **Example**: "How React Reconciliation works."
*   **Rule**: No code steps. Only concepts.

## 4. Reference (Information-Oriented / Theoretical)
*   **Goal**: Accuracy.
*   **Style**: A dictionary.
*   **Structure**: "API Endpoint: /users. Returns JSON."
*   **Example**: API Docs, Configuration Maps.
*   **Rule**: No opinions. Just facts.
